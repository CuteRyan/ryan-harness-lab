# 포트폴리오 프로젝트 정리 스킬

## 목적
현재 프로젝트 디렉토리를 심층 분석하여, 포트폴리오에 넣을 수 있는 **표준화된 MD 문서**를 생성한다.

핵심 가치:
- **"왜(Why)"** — 무엇을 만들었는지보다 **왜 이렇게 설계했는지, 왜 이 기술을 선택했는지, 어떤 접근 방식을 취했고 그 근거는 무엇인지**
- **기술적 깊이** — 표면적 나열이 아닌, 내부 구조·알고리즘·설계 패턴까지 드러내는 수준
- **문제 해결 스토리** — 어떤 문제에 부딪혔고, 어떤 시도를 거쳐, 어떻게 해결했는지의 과정

## Trigger
- `/portfolio` 명령어로 직접 호출
- "포트폴리오 정리해줘", "이 프로젝트 포트폴리오에 넣자" 등

## 실행 프로세스

### 1단계: 프로젝트 심층 파악
아래를 **자동으로** 조사한다. 사용자에게 일일이 묻지 않는다.

1. **프로젝트 구조** — `ls`, Glob으로 디렉토리 구조·모듈 분리 방식 파악
2. **docs/ 문서** — HISTORY.md, 설계 문서, README 등 읽기
3. **코드 심층 분석** — 주요 모듈·클래스·함수 읽기, 설계 패턴 식별, 기술 스택 전체 매핑
4. **아키텍처 파악** — 레이어 구조, 모듈 간 의존성, 데이터 흐름 추적
5. **git 히스토리** — `git log --oneline -50`으로 개발 흐름·의사결정 변천 파악
6. **설정 파일** — package.json, requirements.txt, pyproject.toml, Makefile, CI 설정 등에서 빌드·배포·품질 관리 체계 파악
7. **CLAUDE.md / 프로젝트 메모리** — 기존 결정 사항, 설계 원칙 확인
8. **테스트 구조** — 테스트 전략, 커버리지, 테스트 종류(단위/통합/E2E) 파악

### 2단계: 사용자 인터뷰 (간결하게)
자동 분석으로 파악할 수 없는 것만 묻는다:

- "이 프로젝트의 핵심 동기(왜 만들었는지) 한 줄로 말씀해주세요"
- "가장 자랑하고 싶은 기술적 도전/해결은 무엇인가요?"
- 이미 docs/에 충분한 정보가 있으면 질문 생략 가능

### 3단계: MD 문서 작성
아래 양식으로 작성한다.

### 4단계: 저장
- 저장 위치: `C:/Users/rlgns/OneDrive/문서/포트폴리오/{프로젝트명}.md`
- 파일명: 프로젝트명을 영문 snake_case로 (예: `election_simulator.md`, `btc_futures_bot.md`)

- **결정 사항**: 무엇을 결정했는지
- **맥락**: 어떤 상황/제약 하에서 결정했는지
- **근거**: 왜 이 방향을 선택했는지
- **트레이드오프**: 무엇을 얻고 무엇을 포기했는지

## 작성 규칙

### 톤 & 문체
1. **명사형 종결** — "~구현", "~설계", "~도입", "~해결". 경어체/반말 금지
2. **기술적 깊이 유지** — 표면적 나열("React 사용")이 아닌, 왜 그 기술을 쓰는지, 내부적으로 어떻게 동작하는지까지 서술
3. **전문적이되 읽기 쉽게** — 전문 용어를 자연스럽게 사용하되, 맥락으로 의미가 전달되도록 서술. 약어는 첫 등장 시 풀네임 병기

### 내용 원칙
4. **"왜(Why)"가 모든 섹션의 핵심** — 무엇(What)과 어떻게(How)는 코드를 보면 안다. 왜(Why) 그렇게 했는지가 포트폴리오의 가치
5. **대안 비교 필수** — 기술 선택 시 "A 대신 B를 선택한 이유"를 반드시 포함. 검토 과정 자체가 역량을 보여줌
6. **문제 해결 스토리** — 도전 섹션에서는 "문제 → 분석 → 시도 → 해결 → 인사이트" 흐름을 반드시 유지. 처음부터 성공한 것보다 시행착오 과정이 실력을 증명
7. **정직하게** — 어려웠던 점, 실패한 시도, 트레이드오프를 숨기지 않음. 한계를 인식하고 있다는 것 자체가 역량
8. **사실만** — docs/, git log, 코드에서 확인된 사실만 기재. 추측이나 과장 금지

### 형식 원칙
9. **코드 블록 최소화** — 긴 코드 붙여넣기 금지. 핵심 로직 설명은 텍스트로, 참고용 스니펫은 5줄 이내로 제한
10. **다이어그램은 텍스트 기반** — ASCII/유니코드 다이어그램 사용
11. **분량**: 전체 문서 A4 5~8페이지 수준. 너무 짧으면 깊이 부족, 너무 길면 집중력 분산
