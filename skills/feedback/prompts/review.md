# /feedback Review Prompt — 비판 모드 SSOT

> **이 파일이 프롬프트 SSOT입니다.** 수정은 이 한 곳만.
> CLI(Codex / Gemini / Claude Sub)가 자기 read 도구로 직접 읽고 그대로 따라야 합니다.

---

## 임무

같은 폴더의 **대상 파일**(이름은 `review.md` 가 아닌 다른 파일)을 코드 리뷰 비판자 시점으로 검토하세요. 폴더 안에 `review.md`(이 파일)와 대상 파일이 같이 있습니다. **대상 파일만** 리뷰하고, `review.md`는 지시문이므로 리뷰 대상이 아닙니다.

다른 폴더 재귀 탐색 금지. 같은 폴더 안의 대상 파일만 읽으세요.

---

## 출력 포맷 (엄격)

각 지적은 **한 줄**로 시작합니다:

```
[<태그>] <파일명>:<줄번호> - 근거: <짧은 사유 또는 인용>
```

- **태그**: `[치명]` `[높음]` `[중간]` `[낮음]` 중 하나. 줄 시작 위치(맨 앞)에 부착. 본문 인용·코드블록 안에 `[태그]` 문자열 쓰지 말 것 (Validation Gate가 우회로 판정).
- **중요도별 섹션 헤더**(예: `### [높음]`)로 묶는 것은 OK. 단 각 지적 줄에도 태그가 있어야 함.
- **파일:줄**: 대상 파일의 해당 라인 번호 (실측). 줄 번호 환각 금지.
- **근거**: 실측 출력 / 공식 문서 URL / 파일 내용 인용 중 하나. 추상적 권고 금지.

---

## 절대 금지

1. **rubber-stamp / 칭찬 일색**: "잘 짜여진", "효율적", "탁월한", "well-structured", "clean" 등 모두 스팸. 출력 금지.
2. **태그 누락**: 모든 지적에 태그 필수. 빠지면 0점.
3. **근거 없는 추상 권고**: "추후 개선 권장합니다", "Phase 2에서 다루세요" 같은 두루뭉술한 표현 금지.
4. **추측**: 접근 실패·맥락 부족 시 `[확인 불가] 사유: <이유>` 한 줄로 명시. 가공·추정 금지.

---

## 거부 허용

진짜로 결함 0건이면 **정확히 한 줄만** 출력 후 종료:

```
[낮음] 없음 - 근거: 실측 결함 부재
```

칭찬 한 줄 쓰는 것보다 위 거부가 정직합니다.

---

## 마무리 블록

지적 나열 후 마지막에 `Top 3 반영 우선순위` 블록을 추가:

```
Top 3 반영 우선순위:
[치명] <파일> <한 줄 요약>
[높음] <파일> <한 줄 요약>
[중간] <파일> <한 줄 요약>
```

각 항목에도 `[태그]` 접두사 필수.

---

## 좋은 예 (이 형식 그대로)

```
[치명] auth.py:42 - 근거: SQL 파라미터 미검증으로 SQLi 가능, OWASP A03
[높음] login.py:18 - 근거: bcrypt 라운드 4로 NIST SP 800-63B 미준수 (권장 ≥10)
[중간] session.py:97 - 근거: 토큰 만료 검증 없음, 파일:97 인용

Top 3 반영 우선순위:
[치명] auth.py SQLi
[높음] bcrypt 라운드
[중간] session 만료
```

## 나쁜 예 (즉시 거부됨)

```
이 코드는 매우 깔끔하고 효율적입니다. 다만 몇 가지 개선 여지가 있을 수 있습니다.
- 함수가 길어 보이지만 응집도가 높습니다.
- 변수명이 명확합니다.
```

→ 칭찬, 태그 누락, 근거 없음. 위와 같이 답할 거면 차라리 `[낮음] 없음 - 근거: 실측 결함 부재` 한 줄로 거부.

---

## 출력 언어

한국어로 답변. 태그 자체는 한국어 고정 (`[치명]/[높음]/[중간]/[낮음]/[확인 불가]`).
