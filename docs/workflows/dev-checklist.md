# 개발 체크리스트 워크플로우 (절차 상세)

> **위치**: 이 문서는 `/checklist` 스킬 기반 워크플로의 **상세 절차·예시·트러블슈팅**을 다룬다.
> **SSOT 분리**:
> - 짧은 규칙(강제) → `~/.claude/rules/dev-checklist.md`
> - 스킬 본문(정의) → `~/.claude/skills/checklist/SKILL.md`
> - 긴 절차/예시/FAQ(이 문서) → `docs/workflows/dev-checklist.md`
>
> 이 세 곳은 **중복이 아닌 보완 관계**다. 충돌 발견 시 SKILL.md를 정답으로 본다.

---

## 0. 한눈에 보기 (절차 다이어그램)

```
[사용자 지시]
    ↓
Phase 0: tiny edit 판정 ── (예) ──→ 체크리스트 없이 바로 수정
    ↓ (아니오)
Phase 1: 체크리스트 생성 + 기준선 기록
    ↓ (turn end → 사용자 승인 대기)
Phase 1.5: approved: true 반영
    ↓
Phase 2: 구현 (항목별 체크)
    ↓
Phase 3: 검증 (Read 증거 + git diff 대조)
    ↓
Phase 4: 더블 체크
    ↓
Phase 5: 구조화 보고 (.checklist.md 최종 상태 첨부)
    ↓ (사용자 승인 대기)
Phase 6: .backups/로 이동 + history 갱신
```

---

## 1. Phase 0 — tiny edit 판정 (스킵 여부 결정)

체크리스트 없이 바로 수정해도 되는 "정말 작은" 변경의 정의.

### 1.1 tiny 조건 (전부 충족 시에만 스킵)
- **단일 파일**의 단일 Edit
- old_string과 new_string이 **각각 3줄 이하** + **각각 240자 이하**
- 아래 "blanket 예외 제거 영역"에 해당하지 않음

### 1.2 blanket 예외 제거 영역 (줄 수 무관, 반드시 체크리스트 필요)
- `conftest.py`, `setup.py`
- 공개 API (외부 노출 함수, 엔드포인트, 라이브러리 export)
- DB 스키마 / 마이그레이션 파일
- 권한·보안 설정 (인증, 권한 체크, 비밀키 처리)
- 배포 설정 (systemd unit, Dockerfile, docker-compose, CI 워크플로)
- 테스트 비활성화 (skip / xfail / 주석 처리)
- 데이터 마이그레이션 스크립트

### 1.3 모호하면 체크리스트를 만든다
- "tiny일까 애매하다" 판단이 들면 무조건 체크리스트.
- 체크리스트는 **1분이면 만든다**, 잘못된 변경은 한 시간을 잡아먹는다.

---

## 2. Phase 1 — 체크리스트 생성 + 승인 대기

### 2.1 mode 자동 감지
작업 내용을 보고 mode를 결정한다:

| 패턴 | mode |
|------|------|
| `.py`, `.ts`, `.go` 등 코드 파일만 수정 | `code` |
| `.md`, 문서·위키만 수정 | `doc` |
| 새 함수/모듈 설계, API 계약 정의 | `design` |
| 코드 + 문서, 또는 스킬·훅처럼 양쪽 영향 | `mixed` |

### 2.2 작업 전 기준선
체크리스트 생성 직전에 다음을 실행하고 결과를 그대로 기록:

```bash
git status --short
```

이 출력이 "변화 0의 시작점"이다. Phase 3에서 이 기준선과 비교한다.

### 2.3 체크리스트 작성 후 즉시 멈춤

**규칙**: 체크리스트 작성 후 **구현 파일을 절대 수정하지 않는다**. 승인 요청만 하고 **턴을 종료한다**.

이유:
- 모델이 "이 정도는 명백하니 바로 구현하자"라며 진입하면 Phase 1의 의미가 사라진다.
- 사용자가 체크리스트를 검토할 시간이 0이 된다.
- `approved: false`가 첫 작성 시점의 디폴트인 이유는 **승인 절차를 강제로 두 턴 분리**하기 위해서다.

### 2.4 승인 처리
다음 사용자 메시지에 명시 승인이 있으면(예: "ㄱㄱ", "승인", "진행"), **다음 턴에서**:
- `approved: true`
- `status: approved`
로 변경하고 Phase 2로 진입한다.

치명/높음 항목 변경(범위 확장, 새 파일 추가 등)은 **재승인** 필요.

---

## 3. Phase 2 — 구현

- 체크리스트 항목을 하나씩 처리하며 `[x]` 체크
- 진행 중 예상 외 이슈 발생 → 체크리스트에 항목 추가 (재승인 필요 여부 판단)
- 코드 수정 후 즉시 Phase 3로 가지 않고, **Phase 2 단위로 묶어서 점검**

---

## 4. Phase 3 — 검증 (Read 증거 + git diff 대조)

### 4.1 Read 증거 의무
- "확인했다"는 선언만으로는 **부족하다**.
- 각 항목별로 해당 파일을 Read 또는 Grep하고, 결과의 핵심 라인을 응답에 인용한다.
- 예시:
  ```
  ✅ B-3. tiny edit 의미 조건
  Grep "3줄.*240자" SKILL.md → L132 매치 확인
  ```

### 4.2 git 차이 대조
작업 후 다음을 실행:

```bash
git status --short
git diff --stat
```

이 결과가 체크리스트의 "수정 대상 파일"과 **1:1 일치**해야 한다. 불일치 시:
- 누락 변경 파일 → 체크리스트에 소급 추가
- 의도치 않은 변경 → 즉시 되돌림

### 4.3 mode별 검증

#### mode: code
- ruff / black / eslint 등 프로젝트 린트 통과
- 단위 테스트 실행
- **운영 규모** 통합 테스트 (5건 통과로 끝내지 말 것 — 100건+ DB 저장 확인)
- 사이드 이펙트: 수정 안 한 기존 기능 영향 점검

#### mode: doc
- 연관 문서 목록 작성
- 교차 검증: 동일 개념의 표현 일치, 데이터 흐름 모순 없음
- 스키마/필드명이 실제 코드/DB와 일치
- 한 문서에서 "불필요"라고 한 것을 다른 문서에서 "필수"로 두지 않았는지

#### mode: design
- 입출력 계약 명세 (시그니처 + 의미)
- 실패/에러 케이스 처리 방안
- 트랜잭션 범위 (DB 관련)
- 의존 관계 (순서 중요한 작업)

#### mode: mixed
- 위 모든 항목 + 동기화 책임 명시
- 예: 스테이징(`Harness-engineering/skills/`) → 운영(`~/.claude/skills/`) SHA256 일치 확인

---

## 5. Phase 4 — 더블 체크

Phase 3을 통과해도 **한 번 더** 전체를 점검한다.

### 5.1 체크리스트 자체 검증
- 처음에 놓친 항목이 없는가?
- 요구사항 원문(피드백 문서, 이슈, 사용자 메시지)을 다시 읽고 빠진 것 확인

### 5.2 실행 검증
- 체크만 하고 실제 반영 안 된 것은 없는가?
- Phase 3에서 Read하지 않은 항목이 있다면 **여기서 반드시 Read** 후 증거 남기기

### 5.3 일관성 검증
- 수정 내용이 다른 코드/문서와 여전히 일치하는가?
- 특히 SSOT 관계가 있는 파일들(rules / skills / docs/workflows)

---

## 6. Phase 5 — 보고 (구조화 리포트)

"완료했습니다"만 말하는 것은 **금지**. 아래 구조를 따른다.

### 6.1 필수 포함 항목

```markdown
## 1. 완료 항목
- (파일명:라인 단위로 무엇을 했는가)

## 2. 검증 결과
- 린트: ✅/❌
- 테스트: ✅/❌
- 사이드 이펙트: ✅/❌
- 교차 검증(doc일 때): ✅/❌

## 3. 발견된 이슈
- (없으면 "없음" 명시)

## 4. 미완료/보류 항목
- (없으면 "없음" 명시)

## 5. 대조 결과
- 전체 N개 중 N개 완료
- 불일치: N건 (수정 완료 / 미완료)

## 6. .checklist.md 최종 상태
\`\`\`markdown
(체크리스트 파일 전체를 코드블록으로 첨부 — 체크박스 상태가 보이도록)
\`\`\`
```

### 6.2 코드블록 첨부가 필요한 이유
- "5/5 완료" 같은 요약은 거짓말로 만들기 쉽다.
- 체크리스트 파일 자체를 첨부하면 사용자가 **즉시 검증 가능**.
- 미완료/보류 항목이 있다면 코드블록 안에서 `[ ]`로 그대로 보인다.

## 7. Phase 6 — 정리

### 7.1 보존 정책 (삭제 금지)
- 주인님 승인 후 `.checklist.md`를 **삭제하지 않고 이동**한다.
- 이동 위치: `.backups/.checklist.md.완료_{슬러그}_{YYYY-MM-DD}.md`
- 권장 슬러그: 체크리스트 제목의 핵심 키워드 (예: `feedback태깅강화`, `7건반영`)

### 7.2 history 갱신
- 의미 있는 작업이면 `docs/history/{YYYY-MM-DD}.md`에 entry 추가
- `docs/history/index.md`의 "진행 중" 섹션 관리도 함께

### 7.3 자의적 삭제 금지
- 모델이 "끝났으니 정리하자"며 자의적으로 삭제 금지
- 사용자가 명시적으로 "삭제해" 또는 "정리해"라고 말한 경우에만 이동(삭제 아님)

## 8. `.checklist.md` 템플릿 (mode별)

### 8.1 공통 골격

```markdown
# 체크리스트: [작업명]

---
approved: false
status: draft
mode: code|doc|design|mixed
작성일: YYYY-MM-DD
근거: (피드백 문서 / 이슈 / 사용자 요청 등)
---

## 배경 / 목적
(왜 이 작업을 하는가 — 1~3문장)

## 작업 전 기준선 (git status --short)
\`\`\`
(작업 시작 시점의 git status --short 출력 그대로)
\`\`\`

## 작업 항목
- [ ] 항목 1
- [ ] 항목 2

## 수정 대상 파일
- `path/to/file` — 변경 내용

## 검증 항목
(mode별로 아래 섹션 참조)

## 롤백 계획 (실패 시 복구 방법)

- 복구 절차:
- 보존 정책: 삭제 금지, `.backups/`로 이동

## 더블 체크
- [ ] 체크리스트에 빠진 항목 없는가?
- [ ] 실제 반영 안 된 것은 없는가? (Read 증거 남겼는가?)
- [ ] 다른 코드/문서와 일관성 유지되는가?

## 승인 규칙
- 작성 후 구현 파일 수정 금지, 승인 요청만 하고 턴 종료
- 다음 사용자 메시지의 명시 승인 후 다음 턴에서 approved: true
- 치명/높음 항목 변경은 재승인
```

### 8.2 mode: code 추가 검증 섹션

```markdown
## 검증 항목 (code)
- [ ] 린트 통과 (ruff / black / eslint)
- [ ] 단위 테스트 통과
- [ ] 운영 규모 통합 테스트 (100건+ DB 저장 확인)
- [ ] 사이드 이펙트 없음 (수정 안 한 기능)
- [ ] 작업 후 `git status --short` / `git diff` = 체크리스트 1:1 일치
```

### 8.3 mode: doc 추가 검증 섹션

```markdown
## 검증 항목 (doc)
- [ ] 연관 문서 목록 작성 + 교차 검증 완료
- [ ] 용어/개념 다른 문서와 일치
- [ ] 스키마/필드명 실코드/DB와 일치
- [ ] 링크 유효성
- [ ] 작업 후 `git status --short` / `git diff` = 체크리스트 1:1 일치
```

### 8.4 mode: design 추가 검증 섹션

```markdown
## 검증 항목 (design)
- [ ] 입출력 계약 명세 (시그니처 + 의미)
- [ ] 실패/에러 케이스 처리 방안 명시
- [ ] 트랜잭션 범위 명시 (DB 관련)
- [ ] 의존 관계 다이어그램
- [ ] 기존 코드/스키마와 충돌 없음
```

### 8.5 mode: mixed 추가 검증 섹션

```markdown
## 검증 항목 (mixed)
- [ ] code 섹션 전체
- [ ] doc 섹션 전체
- [ ] 동기화 책임 명시 (스테이징 → 운영 SHA256 일치 등)
```

## 9. 자주 묻는 질문 (FAQ)

### Q1. tiny edit인지 애매할 때는?
체크리스트를 만든다. 30초 손해보고 한 시간 안전을 산다.

### Q2. 작업 도중 새 파일이 추가됐다면?
Phase 3에서 "누락 항목 탐지"로 잡고 체크리스트에 소급 추가한다.
치명/높음 항목 추가라면 사용자에게 재승인 요청.

### Q3. Phase 1에서 승인 없이 진입하면 안 되는 이유?
- 사용자에게 검토 시간이 0이 됨
- 모델이 자기 가정대로 진행 → 의도와 다른 방향으로 갈 위험
- `approved: false` 디폴트는 "두 턴 분리"를 강제하는 안전장치

### Q4. `.checklist.md`를 삭제하면?
하지 말 것. 근거가 사라지면 같은 실수를 반복한다. 항상 `.backups/`로 이동.

### Q5. mode를 바꾸려면?
체크리스트의 frontmatter `mode` 값만 변경. 단, mode 변경은 검증 섹션 재작성을 동반하므로 사실상 재승인이 필요하다.

### Q6. 같은 세션에서 체크리스트가 여러 개 필요하면?
하나의 큰 작업으로 묶을 수 있으면 묶고, 정말 독립적이면 작업이 끝날 때마다 `.backups/`로 이동 후 새로 시작.

## 10. 트러블슈팅

### 10.1 "체크리스트 없이 작업했다"는 사용자 지적
- 즉시 멈추고 체크리스트를 소급 작성한다.
- 이미 변경한 파일을 체크리스트의 "수정 대상 파일"로 기록.
- Phase 3 검증부터 다시 시작 (Read 증거 + git diff 대조).
- 이런 사례는 `docs/feedback/`에 패턴으로 남겨 재발 방지.

### 10.2 SSOT 충돌 발견 (rules vs skills vs workflows)
- 정답은 **SKILL.md**.
- rules는 짧은 규칙 → SKILL.md 요약본
- workflows(이 문서)는 긴 절차 → SKILL.md의 "어떻게" 보강
- 다른 두 곳에 SKILL.md와 모순되는 내용이 있으면 SKILL.md 기준으로 통일.

### 10.3 스테이징/운영 drift
- `Harness-engineering/skills/checklist/SKILL.md` (스테이징)
- `~/.claude/skills/checklist/SKILL.md` (운영)
- 편집 후 SHA256 비교로 drift 확인 의무. 다르면 스테이징을 복사.

### 10.4 git status가 깨끗한데 체크리스트가 있는 경우
- 이전 작업 종결 누락. `.backups/`로 이동시키고 새 작업 시작.
