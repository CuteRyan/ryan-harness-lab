# 개발 체크리스트 워크플로우

> rules/dev-checklist.md의 상세 절차. 규칙(강제)은 rules/에, 절차(어떻게)는 여기에.

---

## Phase 1: 체크리스트 생성 (코드 수정 전)

1. **`.dev-checklist.md` 생성** — 프로젝트 루트에 배치
2. 포함 내용:
   - [ ] 구현 항목 (무엇을 만드는가)
   - [ ] 수정 대상 파일 목록
   - [ ] 검증 항목 (어떻게 확인하는가)
   - [ ] 더블 체크 항목
3. **주인님 승인** 후 코드 수정 진행

## Phase 2: 구현 (체크리스트 따라 진행)

- 각 항목 완료 시 `[x]`로 체크
- 예상 외 이슈 발생 시 체크리스트에 항목 추가

## Phase 3: 구현 대조 검증 (코드 vs 체크리스트)

> 체크만 하고 넘어가는 것을 방지. **실제 파일을 열어 확인**해야 한다.

1. **항목별 1:1 대조** — 체크리스트의 각 구현 항목에 대해:
   - 해당 파일을 **직접 Read**하여 구현이 반영되었는지 확인
   - 체크리스트에 적은 의도와 실제 코드가 일치하는지 비교
   - 불일치 발견 시 즉시 수정 (수정 후 다시 대조)
2. **누락 항목 탐지** — 구현 과정에서 추가로 변경한 파일 확인
3. **Ruff 린팅 통과**
4. **관련 기능 직접 테스트** — 소규모가 아닌 운영 규모로
5. **사이드 이펙트 확인** — 수정하지 않은 기존 기능이 깨지지 않았는지

## Phase 4: 더블 체크

1. **체크리스트 자체 검증** — 처음에 놓친 항목이 없는가?
2. **실행 검증** — 체크만 하고 실제 반영 안 된 것은 없는가?
3. **일관성 검증** — 수정한 내용이 다른 코드/문서와 여전히 일치하는가?

## Phase 5: 피드백 리포트 (주인님께 보고)

리포트 필수 포함 항목:
1. **구현 완료 항목** — 무엇을 만들었는가 (파일명:라인 포함)
2. **검증 결과** — 린팅, 테스트, 사이드 이펙트 각각의 통과/실패 여부
3. **발견된 이슈** — 없으면 "없음" 명시
4. **미완료/보류 항목** — 없으면 "없음" 명시
5. **체크리스트 대조 결과** — 전체 N개 중 N개 완료, 불일치 N건

## Phase 6: 정리

- **주인님이 리포트 확인 후 승인**해야 `.dev-checklist.md` 삭제 가능
- HISTORY.md 업데이트 (필요 시)

---

## 템플릿

```markdown
# 개발 체크리스트: [작업명]

## 구현 항목
- [ ] 항목 1
- [ ] 항목 2

## 수정 대상 파일
- `path/to/file.py` — 변경 내용

## 구현 대조 검증 (Phase 3)
- [ ] `파일1.py` — Read로 확인, 의도대로 구현됨
- [ ] 누락 변경 파일 없음 확인

## 검증 항목
- [ ] 린팅 통과
- [ ] 기능 테스트 (운영 규모)
- [ ] 기존 기능 영향 없음

## 더블 체크 (Phase 4)
- [ ] 체크리스트에 빠진 항목 없는가?
- [ ] 실제 반영 안 된 것은 없는가?
- [ ] 다른 코드와 일관성 유지되는가?

## 피드백 리포트 (Phase 5)
- **구현 완료**: (파일명:라인 목록)
- **검증 결과**: 린팅 ✅/❌ | 테스트 ✅/❌ | 사이드이펙트 ✅/❌
- **발견 이슈**: (없으면 "없음")
- **미완료/보류**: (없으면 "없음")
- **대조 결과**: 전체 N개 중 N개 완료, 불일치 N건
```
