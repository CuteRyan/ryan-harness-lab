# Gemini 피드백 — 세션 인계 / project-history 스킬 확장 설계

## 검토 대상
- project-history 스킬에 세션 간 핸드오프 기능 통합 설계안
- 핵심: 마이그레이션 → index.md 최상단 진행 중 섹션 → 일별 동기화 → SessionStart 훅

## 원문

### 1. 설계의 약점 및 놓친 시나리오
- **Git 브랜치 불일치**: `index.md`의 '다음 작업'과 실제 작업 중인 `Git Branch`가 다를 때 인지 부조화가 발생할 수 있음. 핸드오프 섹션에 **현재 Git 커밋 해시나 브랜치명**을 포함해야 함.
- **컨텍스트 오버플로**: 세션이 길어질수록 `SessionStart`로 주입된 정보가 뒤섞여 에이전트가 혼란을 느낄 수 있음.

### 2. 이중 기록 동기화 해결 (SSOT 전략)
- **역방향 관리(Archive 방식)**: 일별 파일에서 `index`로 데이터를 올리는 '복제' 대신, `index.md` 상단의 '진행 중' 섹션을 **유일한 작성처(SSOT)**로 삼을 것. 작업 완료 시점에만 해당 내용을 일별 파일로 **이동(Cut & Paste)** 시키면 동기화 오류가 원천 차단됨.

### 3. SessionStart 훅의 필요성
- **조건부 실행 권장**: 모든 시작 시 주입은 토큰 낭비. 마지막 수정 후 일정 시간(예: 1시간)이 지났거나, 사용자가 `claude resume` 같은 특정 명령으로 진입할 때만 작동하게 할 것. 평소에는 수동 호출이 컨텍스트 제어권 측면에서 유리.

### 4. 마이그레이션 순서
- **섹션 추가 먼저**: 현재 154줄은 단일 파일로도 충분히 가벼움. 우선 `HISTORY.md` 상단에 섹션을 운영하며 **'핸드오프가 실제로 잘 워킹하는지'** 1~2주 검증한 뒤 폴더 구조로 마이그레이션하는 것이 훨씬 안전함. 구조 변경은 비용이 듦.

## 핵심 지적
1. **Git 브랜치/커밋 해시 포함** 필요 — 핸드오프와 실제 git 상태 불일치 방지
2. **역방향 SSOT (Archive)**: index 상단=SSOT, 완료 시 일별로 cut & paste → 이중 기록 원천 차단
3. **SessionStart 훅 조건부 실행** (시간 경과 / resume 명령 시만)
4. **마이그레이션은 1~2주 검증 후로 미루기** — 154줄은 단일 파일로 충분
