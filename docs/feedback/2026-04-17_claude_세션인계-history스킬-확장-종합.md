# 종합 — 세션 인계 / project-history 스킬 확장 설계

## 검토 대상
세션 간 작업 인계 강화를 위한 project-history 스킬 확장 설계안 (마이그레이션 → index 최상단 진행 중 섹션 → 일별 동기화 → SessionStart 훅).

## Codex 요약 (날카로운 리뷰)
1. **이중 기록 = 신뢰 붕괴**: index를 "거울"로 두면 두 개의 원본이 됨 → index는 SSOT가 아닌 **추출 결과/캐시**여야 함 (일별 파일이 SSOT)
2. **SessionStart 훅 얇게 + 조건부**: handoff 5줄 + git status/log만, 깊은 탐색은 수동 `/project-history`
3. **마이그레이션 후순위**: 단일 HISTORY.md 상단 섹션으로 1~2주 검증 후 폴더 이전

## Gemini 요약 (넓은 리뷰)
1. **Git 브랜치/커밋 해시 포함** 필요 — 핸드오프와 실제 git 상태 불일치 방지
2. **역방향 SSOT (Archive 방식)**: index 상단을 **SSOT**로 두고, 완료 시 일별 파일로 cut & paste 이동 → 동기화 오류 원천 차단
3. **SessionStart 훅 조건부 실행** (시간 경과 / resume 명령 시만)
4. **마이그레이션 1~2주 검증 후로**

## 공통 지적 (둘 다 언급, 반드시 반영)
| # | 지적 | 합의 강도 |
|---|------|----------|
| 1 | **이중 기록 동기화는 위험** — 어느 한쪽이 SSOT여야 함 | 강 (해법은 다름) |
| 2 | **SessionStart 훅은 얇게/조건부** — 무조건 주입은 비효율 | 강 |
| 3 | **마이그레이션 미루고 단일 HISTORY.md 상단 섹션부터** | 매우 강 (둘 다 명시적으로 동의) |

## 상이한 점 — SSOT 위치
- **Codex**: 일별 파일 = SSOT, index = 추출/캐시 (읽기 전용 렌더링)
- **Gemini**: index 상단 = SSOT, 완료 시 일별로 archive (Cut & Paste)

### 판단
**Gemini의 역방향 SSOT가 더 깔끔함**. 이유:
- 진행 중 항목은 항상 한 곳(index 상단)에서만 편집 → 편집 충돌 없음
- 완료 → cut → paste 한 번이면 끝, 추가 동기화 코드 불필요
- Codex 방식은 "추출/렌더링" 로직을 또 만들어야 함 (스킬 구현 부담)
- 다만 지금 단계에서는 단일 HISTORY.md 상단에 섹션을 두는 것이라 SSOT 논의는 마이그레이션 단계로 유보 가능

## 반영 우선순위
| 순위 | 항목 | 반영 대상 |
|------|------|----------|
| 1 | 마이그레이션 미루기 — 단일 HISTORY.md 상단에 "🔄 진행 중" 섹션부터 추가 | docs/HISTORY.md, project-history 스킬 update 동작 |
| 2 | 핸드오프 섹션에 **git branch + 마지막 커밋 해시** 포함 | 섹션 템플릿 |
| 3 | SessionStart 훅 — 조건부 + 얇게 (handoff 섹션 + git status --short + git log -5) | settings.json |
| 4 | 1~2주 운영 후 폴더 마이그레이션 + SSOT 위치 결정 (Gemini 방식 유력) | Phase 2 |

## 미반영/유보
- **"resume 명령 시만 훅 작동"**: Claude Code에 resume 진입 구분 신호가 있는지 확인 필요 — 없으면 시간 기반 조건만 사용

## 최종 결정 (2026-04-17)

### SSOT 위치: **Gemini의 역방향 SSOT 채택 — index 상단 = 원본**

**근거**:
| 항목 | Codex (일별=SSOT) | Gemini (index=SSOT) |
|------|------------------|---------------------|
| 동기화 코드 | 필요 (추출/렌더링) | 불필요 (cut & paste) |
| SessionStart 훅 | 최신 일별 찾기 + 섹션 파싱 (2단계) | index 상단 N줄 읽기 (1줄) |
| 편집 위치 | 일별 + 자동 추출 신뢰 | 한 곳만 |
| 버그 가능성 | 추출 로직 = 깨질 수 있음 | 사람 손 작업 = 단순 |

핵심: **추출 로직은 코드이고, 코드는 버그가 난다**. 하네스 "코드 줄이기" 원칙과 일치.

### Gemini 안의 약점 + 보완
- **약점**: 여러 작업 동시 진행 시 index 최상단 누적 → 시작일 식별 곤란
- **보완**: 각 항목에 시작일 + 선택적 git 메타 표기

```markdown
## 🔄 진행 중 (다음 세션 인계)
- [2026-04-17] 작업명 / 다음 단계 / 미결 (브랜치: main, 커밋: a1b2c3)
- [2026-04-15] 작업명 / ...
```

### 단계별 적용
- **Phase 1 (지금, 단일 HISTORY.md)**: SSOT 논의 무의미 — 어차피 한 파일. 상단에 "🔄 진행 중" 섹션 추가만. 1~2주 운영
- **Phase 2 (마이그레이션 후)**: index.md 상단 = SSOT 확정, 일별 파일은 "완료 아카이브" 역할로 분리. 작업 완료 시 cut & paste

## 결론
원안의 순서가 잘못됨. **단일 HISTORY.md 상단에 섹션부터 추가하고 1~2주 검증 → 폴더 마이그레이션 + Gemini의 역방향 SSOT 적용**이 안전한 경로. SessionStart 훅은 Phase 1에서는 수동 호출만, 패턴 안정화 후 자동 주입 도입.

# 2차 검토 결과 (최종안 재검토)

## Codex 2차 요약
1. **Phase 1 관성 위험** — 단일 파일 습관이 Phase 2까지 이어져 사람 기억 의존
2. **전환 기준은 기간이 아닌 신호 기반**: "진행 중 3개 이상" 또는 "인계 누락 1회"
3. **누적 방지 자동화 필요**: `/project-history`가 14일 초과 항목 경고
4. **형식**: `[시작일][상태][branch@sha] 작업 / 다음행동 / blocker`
5. **Phase 1 훅 제외 OK** — 단, 수동 호출 여부 + 복구 성공 여부 기록해야 검증 가능

## Gemini 2차 요약
1. **Phase 1부터 엄격한 섹션/형식 유지** — Phase 2 자동화 기술 부채 최소화
2. **시각적 한계 (7개 초과 강제 아카이빙)** — 심리적 압박이 자동화보다 효과적
3. **형식**: `[시작일] 상태 | 작업명 | 다음 행동(동사 시작)`. Git 메타는 **대형 작업에만 선택적**
4. **훅 제외 합리적** — 도구 의존 전 "기록 문화" 검증 필터

## 공통 합의 (둘 다 강조)
| # | 합의 사항 |
|---|----------|
| 1 | **누적 방지 임계값 명문화 필요** (Codex: 14일, Gemini: 7개) — 자동화/수동 모두 OK |
| 2 | **Phase 1부터 엄격한 형식 유지** — 부채 회피 |
| 3 | **Phase 1 훅 제외 OK** |
| 4 | **시작일 + 상태 메타 필수** |

## 상이점 해소
| 항목 | Codex | Gemini | 채택 |
|------|------|--------|------|
| 임계값 | 14일 초과 | 7개 초과 | **OR 통합**: 7개 초과 또는 14일 초과 시 경고 |
| Git 메타 | 형식에 박음 | 대형 작업만 선택 | **Gemini 채택** — 매번 hash 적기는 피로 |
| Phase 1 검증 | 호출/복구 기록 | 기록 문화 검증 | **Gemini 채택** — 메타 작업 부담 회피, 검증 종료 시 한 번 회고 |

## 최종 결정 (2차 반영, 2026-04-17)

### 항목 형식 (통합안)
```markdown
## 🔄 진행 중 (다음 세션 인계)
- [2026-04-17] 진행 중 | HISTORY 인계 섹션 추가 | 다음: 스킬 update 동작 수정 | 미결: 없음
- [2026-04-15] 보류 | 대형 리팩터링 (브랜치: feat/x, 커밋: a1b2c3) | 다음: 테스트 추가 | 미결: 스키마 결정
```
- **필수**: `[시작일] 상태 | 작업명 | 다음(동사) | 미결`
- **선택**: 대형/장기 작업에만 `(브랜치: x, 커밋: hash)` 추가

### 누적 방지 규칙
- **시각적 한계**: 진행 중 항목 **7개 초과** 시 즉시 정리 (Gemini)
- **시간 한계**: 항목당 **14일 초과** 시 `/project-history`가 경고 (Codex)
- 두 조건은 OR — 하나라도 걸리면 정리

### 전환 기준 (Phase 1 → Phase 2)
다음 중 하나 발생 시:
- 진행 중 항목이 일상적으로 3개 이상 운영됨 (복잡도 임계)
- 인계 누락 1회 발생 (실패 신호)
- 1주 안정 운영 통과 후 회고에서 "패턴 잡힘" 합의

### Phase 1 검증 방식
- 수동 호출 메타 기록 X (부담)
- 1주 후 회고 1회: "핸드오프가 실제로 다음 세션에 도움됐는가" 평가
- 도움 안 됐으면 형식/위치 재설계, 도움 됐으면 Phase 2 진입

### Phase 1 형식 엄격 유지 (Gemini 1번 지적 반영)
- 단일 HISTORY.md 상단에 추가하더라도 **Phase 2와 동일한 섹션명/형식** 사용
- 마이그레이션 시 그대로 옮길 수 있도록
