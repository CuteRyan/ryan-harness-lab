# Gemini 리뷰 — /checklist 스킬 5차

> 리뷰 대상: `skills/checklist/SKILL.md`
> 관점: 자율적 모델의 '탈출구(Loopholes)' 차단 + 운영 효율성

## 1. 작동성 및 구조적 약점

### 문제: "승인(Approval)" 프로세스의 교착 상태(Deadlock)
- Phase 1과 Phase 6의 "주인님 승인 후" 단계가 **비대화형 환경 원칙**과 충돌. 에이전트가 승인을 기다리며 멈추거나, 반대로 이 구절을 근거로 워크플로우를 아예 무시할 위험.
- **개선 제안**: "승인"을 **"명시적 계획 선언(Explicit Plan Declaration)"**으로 대체
  - 수정 전: "주인님 승인 후 구현 시작"
  - 수정 후: "체크리스트 파일 생성 후, 수행할 작업을 요약 보고하고 즉시 Phase 2로 진입 (사용자가 중단하지 않는 한 진행)"

### 문제: Phase 3의 "100건+ 테스트" 경직성
- 단순 버그 수정/오타 수정에서도 100건 이상 요구하면 모델이 이를 무시 → 전체 스킬 무시로 이어짐.
- **개선**: "작업 성격에 따른 규모 가변제" 도입 (UI/단순로직은 시나리오 기반, 데이터 처리/엔진은 100건+ 샘플링)

## 2. Tiny Edit 기준 (3줄 AND 240자)

- 기준이 지나치게 엄격. Python에서 `import` 한 줄 + 함수 호출 하나만 추가해도 3줄 넘음. `AND` 조건 탓에 "1줄 300자"는 예외가 안 되어 혼란.
- **개선**: **"5줄 이하 OR 500자 이하"**로 완화하거나 **"의미적 단위(Semantic Unit)"** 기준 추가
  - 추천: "기존 로직의 흐름을 바꾸지 않는 단순 값 변경, 주석 수정, 오타 수정은 예외"

## 3. 스킬 통합 vs 분리

- **통합 유지가 압도적으로 유리**. 내부 템플릿 분기로 해결. `/checklist-code`와 `/checklist-doc` 사이 고민은 '결정 피로' 유발.
- **개선**: 스킬 내용 내 `## 템플릿 선택 가이드`를 넣어 모델이 상황에 맞는 항목을 스스로 골라 쓰게 함.

## 4. 빠진 항목 / 과한 항목

### [빠진 항목 — 필수 추가]
1. **롤백 계획 (Rollback Plan)** — Phase 1 체크리스트에 "실패 시 복구 방안(예: `git checkout .`)" 항목 포함. 하네스 '안전' 철학에 부합.
2. **도구 명시 (Tool Usage Enforcement)** — Phase 3 검증에서 `grep`, `read_file` 등 실제 도구로 "눈으로 확인했음"을 **증명**하도록 강제. (환각 방지)

### [과한 항목 — 제거/수정]
1. **Phase 6 `.dev-checklist.md` 즉시 삭제** — 추적성(Traceability) 상실.
   - 수정안: 작업 완료 리포트에 체크리스트 내용 포함 후 삭제 / `.backups/` 폴더로 이동

## 종합 수정 제안

```markdown
### Phase 1: 체크리스트 생성 및 선언
1. `.dev-checklist.md`를 프로젝트 루트에 write_file로 생성.
2. [필수 항목]: 배경, 수정 대상, 검증 시나리오, 롤백 방법.
3. 생성 직후 "체크리스트에 따라 작업을 시작합니다"라고 선언하고 Phase 2 진입.

### Phase 3: 실측 검증 (Empirical Verification)
- "수정 완료" 선언 전, read_file 또는 run_shell_command를 통해
  체크리스트의 각 검증 항목이 실제 파일에 반영되었는지 로그로 증명해야 함.
  (단순히 "확인했습니다"라고 말하는 것은 금지)

### Phase 5: 피드백 리포트
- 작업 요약과 함께 .dev-checklist.md의 최종 상태를 코드블록으로 출력하여
  작업의 완결성을 입증.
```

## 결론

현재 생성본은 설계가 매우 탄탄하지만, **"모델의 자율적 완결성"**을 위해 비대화형 환경에 맞는 '자기 선언적' 구조로 소폭 수정하면 완벽할 것으로 보임.
