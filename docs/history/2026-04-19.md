## 3. Codex 실패 원인 진단

**처음 가설**: PowerShell Constrained Language Mode가 원인

**실측 결과** (직접 PS 테스트):
- `$ExecutionContext.SessionState.LanguageMode` = **FullLanguage** (정상)
- `__PSLockdownPolicy` = 미설정
- WDAC Usermode CI = 0 (Off)
- Smart App Control = Off
- Codex.exe 서명 = Valid (OpenAI)
- 직접 `powershell.exe -NoProfile -Command "[Console]::OutputEncoding=..."` = **성공**

→ **시스템 정책은 무죄**. 문제는 Codex 서브프로세스 내부에서만 발생.

**진짜 범인** — 영문 경로 `C:\harness-test\`에서 Codex 실측 결과:
- **Codex 정상 작동** (proposal.md 요약 성공, 8584 토큰)
- `PropertySetterNotSupportedInConstrainedLanguage` 에러는 여전히 발생하지만 **치명적 아님** — Codex가 매 PS 명령에 붙이는 `[Console]::OutputEncoding=...` prefix만 실패하고 본 명령은 세미콜론 뒤에서 정상 실행
- 한글 파일 내용이 1차에서 CP949로 깨져도 `-Encoding UTF8` 재시도로 해결

**결론**: **"Codex가 한글 경로 파일을 읽을 때 UTF-8 재시도를 위해 복잡한 우회책(cmd/python/pipeline)을 쓰는데, 이 복잡한 패턴들이 `--sandbox read-only` 정책에 막혀서 무한 실패 루프"**가 진짜 원인. 영문 경로에서는 단순 `Get-Content -LiteralPath -Encoding UTF8` 재시도만으로 해결되므로 정책 충돌 없음.
