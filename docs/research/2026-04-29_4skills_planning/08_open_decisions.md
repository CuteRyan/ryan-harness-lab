### 결정 1: /todo → /checklist 트리거 방식 (자동 vs 수동)

**배경**

Gap 1에서 지적했듯 01/05/06 세 산출물이 각각 "선택사항/자동/수동"으로 서술한다. 구현 전에 반드시 확정 필요.

**옵션**

| 옵션 | 설명 |
|------|------|
| A: 수동 | /todo와 /checklist는 완전 독립. 작업 시작 시 주인님이 명시적으로 /checklist를 호출한다. |
| B: 반자동 | /todo에서 특정 항목을 선택하면 "지금 시작하시겠습니까? → /checklist [항목명]을 호출하시려면 '시작'이라고 하세요" 안내 |
| C: 자동 | /todo add 시 자동으로 /checklist 생성 시작 (현재 설계와 불일치, 01_todo_spec.md 역할 정의 위반 가능) |

**Trade-off**

- 옵션 A: 스킬 독립성 최대. 흐름 단순. 단, 주인님이 /todo 등록 후 /checklist 없이 바로 구현을 시작하는 위험 방지 안 됨.
- 옵션 B: 사용자 경험 개선. /checklist 누락 방지. 구현 복잡도 소폭 증가.
- 옵션 C: 01_todo_spec.md L10-12 역할 정의("미래의 할 일 백로그 관리자")와 충돌. 등록 즉시 실행을 요구하는 것은 "나중에 할 것"이라는 정체성과 모순. 기각 권장.

**critic 추천: 옵션 A (수동), Phase 2에서 B 검토**

근거: 하네스 엔지니어링 원칙 중 "자동화는 필요성이 실증된 이후에 도입". 현재 /todo와 /checklist 모두 신규 스킬로 실사용 데이터가 없다. 수동으로 시작해 주인님이 "매번 /checklist 호출이 귀찮다"는 피드백을 줄 때 B로 업그레이드하는 것이 drift 없이 안전하다. 옵션 C는 역할 정의 위반으로 배제.

근거 강도: **중** (원리 기반 — 실사용 데이터 없음)

### 결정 2: index.md 진행 중 섹션 처리 (옵션 A/B/C)

**배경**

04_history_delta.md §3에서 옵션 A(제거)/B(자동 동기화)/C(최소 유지) 세 가지 trade-off를 분석했고 critic의 최종 판단을 명시적으로 요청했다. 현재 index.md 진행 중 5개 항목 중 "종결" 3개가 방치된 사실이 수동 관리의 실패를 입증한다.

**옵션 요약 (04_history_delta.md §3에서 재인용)**

| 옵션 | 핵심 | 장점 | 위험 |
|------|------|------|------|
| A (제거) | 진행 중 섹션 폐지, HANDOFF.md가 전담 | index.md 단순화, 수동 정리 부담 0 | /handoff 미호출 시 인계 정보 완전 소실 |
| B (자동 동기화) | 진행 중 섹션 유지, /handoff가 자동 업데이트 | SSOT 보존, fallback 유지 | 두 파일 간 SSOT 혼란, 구현 복잡 |
| C (최소 유지) | 진행 중 섹션을 포인터로 경량화 | index.md 변경 최소 | 기존 7개 한계 룰 유명무실, 두 파일 모두 읽어야 함 |

**Trade-off 추가 분석**

옵션 A의 "인계 정보 완전 소실" 위험은 Gap 2(세션 비정상 종료 시 HANDOFF.md 자동 생성 미결)와 직접 연결된다. Gap 2가 해결되지 않은 상태에서 옵션 A를 선택하면 fallback이 0이 된다. 현재 진행 중 5개 항목 중 3개가 "종결" 상태임은 옵션 B(자동 동기화)도 유지보수 부담을 낮추지 않는다는 증거다 — 자동화는 "추가"만 하고 정리는 여전히 수동이 필요하기 때문.

**critic 추천: 옵션 C (최소 유지), 단 즉시 정리 선행**

근거 2단계:
1. Gap 2(세션 비정상 종료 복원)가 해결되기 전까지 옵션 A는 시기상조. fallback 경로가 필요하다.
2. 옵션 B는 /handoff SKILL.md가 확정되지 않은 상태에서 "자동 동기화 의무"를 SKILL.md에 박으면 미래 설계에 선제 제약을 건다. 피해야 한다.
3. 옵션 C로 전환하되, 즉시 실행할 것이 있다: 현재 "종결" 3건(2026-04-22, 2026-04-28, 그리고 2026-04-23 일부)을 지금 index.md에서 삭제한다. 이것은 어떤 옵션을 선택해도 해야 할 정리다.

구체 절차: 현재 5개 항목 중 "종결" 마킹 3건 삭제 → 남은 2건("진행 중", "부분 종결")을 HANDOFF.md 초기 내용으로 이전 → index.md 진행 중 섹션 상단에 "상세: HANDOFF.md 참조" 1줄 추가.

근거 강도: **중** (원리 기반 — Gap 2 미해결이 전제 조건)

### 결정 3: [2026-04-23] ④ Gemini rubber-stamp 이월 종결 여부

**배경**

2026-04-23 Day 10 Part 2에서 "Gemini rubber-stamp 성향 자체는 잔존"이라고 이월 ④ 등록. 원래 계획: (a) `run-gemini.ps1` CLI 전용 강경 프롬프트, (b) Gemini 교체, (c) 3 CLI 유지하되 Gemini 가중치 축소. 이후 Day 14에서 5게이트 + 외부 훅(3단계)이 신설됐다.

**Day 14에서 달라진 것 (2026-04-28.md §5, §9 실측)**

| 변경 | ④에 대한 영향 |
|------|--------------|
| B 방식 (prompts/review.md SSOT) | Gemini가 프롬프트 파일을 직접 읽어 형식 준수 100% (2026-04-28.md §4 H 결과: 3/3 VALID) |
| 5게이트 강제 (Step 3) | 메인 Claude가 Gemini 출력을 종합할 때 sycophancy를 자기검열 의무로 포착 |
| 3단계 훅 (feedback-sycophancy-check.py) | 7 카테고리 자동 검출 — "약한 반박", "약한 비판" 카테고리가 Gemini rubber-stamp 패턴을 직접 포착 |

**판정 기준**

이월 ④의 원래 목표: "Gemini rubber-stamp 성향으로 인한 교차검증 다양성 손실을 구조적으로 방어". Day 14 변경들이 이 목표를 달성했는가?

**critic 판정: 종결 권고**

근거 3단계:

1. **B 방식 효과 (강)**: 2026-04-28.md §4 H 결과 — "3/3 VALID"는 이전 "항상 1~2/3"에서의 전환. Gemini rubber-stamp의 형식 우회가 review.md를 직접 읽음으로써 차단됐다.

2. **3단계 훅 (강)**: feedback-sycophancy-check.py의 "약한 반박" / "약한 비판" 카테고리(2026-04-28.md §9.1 결정 2)는 Gemini rubber-stamp 출력의 핵심 패턴(장점 나열, 반박 약함)을 직접 타격한다. dogfood 6/6 PASS 실측 완료(§9.3).

3. **남은 한계 (중)**: 2026-04-28.md §9.5에서 솔직히 인정 — "누락/충돌 정밀도 낮음: LLM 호출 없이 키워드·정규식으로는 한계." Gemini가 rubber-stamp를 다른 단어로 표현하면 false negative 가능성 상존. 그러나 이것은 ④의 원래 문제("다양성 손실")와 다른 차원이다. ④는 "Gemini가 아예 비판을 안 한다"였고, 현재는 "비판하지만 정밀도가 낮을 수 있다"로 수준이 달라졌다.

따라서 ④ 원래 문제(Gemini rubber-stamp로 인한 3 CLI 다양성 손실)는 Day 14 변경으로 구조적으로 흡수됐다고 판단한다. "정밀도 향상"은 별개 문제로 /feedback 스킬 운영 중 실사용 데이터가 쌓이면 자연스럽게 시드 보강으로 처리된다 — 별도 이월 항목이 필요 없다.

**종결 권고 조건**: 2026-04-23 index.md 항목 `[2026-04-23] 부분 종결 | Day 10 이월 ④ Gemini rubber-stamp 재평가`에서 ④를 "Day 14 흡수 완료"로 마킹 후 정리.

근거 강도: **강** (2026-04-28.md §4·§9 실측 인용, dogfood 6/6 PASS 직접 확인)
