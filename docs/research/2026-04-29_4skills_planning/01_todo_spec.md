## 1. 역할 정의

**핵심 역할**: 미래의 할 일 백로그 관리자.

현재 세션과 무관하게 **상시 유지되는** 항목 목록을 관리한다. `/checklist`가 "지금 이 작업"을 관리한다면, `/todo`는 "나중에 할 것"을 관리한다.

비유: `.checklist.md`는 오늘의 작업 지시서, `.todo.md`는 냉장고 화이트보드 메모.

## 2. 트리거 정의

### 자연어 트리거
| 패턴 | 예시 | 동작 |
|------|------|------|
| "투두 추가" / "할 일 추가" | "나중에 /handoff 스킬 만들어야 하는 거 투두 추가해줘" | 항목 추가 |
| "이거 나중에 해줘" | "이거 나중에 해줘" | 항목 추가 |
| "투두 보여줘" / "할 일 목록" | "현재 투두 뭐 있어?" | 목록 조회 |
| "이거 완료" / "이거 지워줘" | "투두에서 /handoff 완료 처리해줘" | 항목 완료 처리 |
| "투두 정리해줘" | "투두 오래된 것들 정리해줘" | 만료 항목 아카이브 |

### 직접 호출
- `/todo` — 현재 목록 조회
- `/todo add [내용]` — 항목 추가
- `/todo done [번호 또는 키워드]` — 완료 처리
- `/todo archive` — 완료·만료 항목 정리

## 3. 파일 위치 결정

**결정**: 프로젝트 루트 `.todo.md` [근거 강도: 중]

**근거**:
- `/checklist`의 `.checklist.md`와 동일한 패턴 — 루트 레벨 숨김파일로 통일 (checklist/SKILL.md L165: `{프로젝트 루트}/.checklist.md`)
- `docs/`는 완료·확정된 결과물 보관소. 진행형 백로그는 어울리지 않는다.
- 글로벌 메모리(`memory/*.md`)는 세션 간 포인터용. 항목 목록 자체를 넣으면 memory-architecture.md 원칙 위반 ("내용 중복 금지")
- `.gitignore`에 추가 권장 (개인 백로그이므로)

**파일 포맷**:
```markdown
# .todo.md

<!-- created: YYYY-MM-DD | project: {프로젝트명} -->

## 백로그
- [ ] #001 항목 내용 (added: 2026-04-29, priority: high)
- [ ] #002 항목 내용 (added: 2026-04-29, priority: normal)

## 완료 (최근 10개)
- [x] #000 이전 항목 (done: 2026-04-28)
```

## 4. 다른 스킬과의 책임 경계

### vs /checklist [근거 강도: 강]

| 구분 | /todo | /checklist |
|------|-------|------------|
| 시제 | 미래 ("나중에 할 것") | 현재 ("지금 하는 것") |
| 파일 | `.todo.md` (상시 유지) | `.checklist.md` (작업 단위 생성·소멸) |
| 승인 흐름 | 없음 (추가/완료 즉시) | 필수 (`approved: false` → 주인님 승인) |
| 완료 시 | 목록에서 완료 마킹 후 보존 | `.backups/`로 이동 (SKILL.md L65-66) |
| 수명 | 프로젝트 존속 기간 내내 | 단일 작업 사이클 |

**경계 규칙**: `/checklist` 생성 시 해당 항목을 `/todo`에서 "진행 중"으로 상태 변경하는 연계는 선택사항 (Phase 2에서 결정).

### vs /handoff [근거 강도: 중]

| 구분 | /todo | /handoff |
|------|-------|----------|
| 발생 시점 | 상시 (언제든 추가 가능) | 세션 종료 시점에만 |
| 목적 | 백로그 관리 | 다음 세션 인계서 |
| 미완 항목 | 영구 백로그 → 완료까지 보존 | 다음 세션에 넘겨야 할 "지금 하다 멈춘 것" |
| 소멸 조건 | `done` 명령 또는 아카이브 | 다음 세션 시작 시 소멸 (인계 완료) |

**핵심 차이**: `.todo.md`의 미완 항목은 "언제 할지 모르는 것". `HANDOFF.md`의 미완 항목은 "지금 세션에서 진행하다가 못 끝낸 것" — 시급도와 맥락이 다르다.

## 5. TaskCreate 호출 정책

**결정**: 사용자 명시 요청 시에만 [근거 강도: 약]

기본 동작: `.todo.md`만 수정. TaskCreate는 별도로 호출하지 않는다.

**이유**: TaskCreate는 팀/에이전트 작업 조율용. 개인 백로그와 혼용하면 TaskList가 오염된다. 사용자가 "팀 작업으로 등록해줘"라고 명시할 때만 TaskCreate 병행.

---

## 6. 완료 시 동작

1. `.todo.md`의 해당 항목 `[ ]` → `[x]` + `(done: YYYY-MM-DD)` 기록
2. 완료 항목은 "완료 (최근 10개)" 섹션으로 이동 보존 (삭제 금지 — 감사 추적)
3. `/history` 자동 기록: 선택사항. 사용자가 "히스토리에 기록해줘"라고 요청할 때만.
4. 10개 초과 시 오래된 완료 항목 → `.backups/.todo.done.{YYYY-MM-DD}.md`로 이동
