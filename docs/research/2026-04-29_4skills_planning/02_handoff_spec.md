## 1. 역할 정의

**핵심 역할**: 세션 종료 시 다음 세션을 위한 인계서 생성/조회.

현재 세션에서 "하다가 멈춘 것"과 "다음 세션이 바로 시작할 수 있는 상태"를 만드는 도구다. 세션이 단절되는 순간에만 필요하며, 인계 완료 후에는 소멸한다.

비유: 교대 근무 인계 노트. 퇴근할 때 다음 사람을 위해 쓰고, 출근한 사람이 읽고 나면 버린다.

- `/handoff done` — 인계서 소멸 (다음 세션이 확인 완료 후)

## 3. 파일 위치 결정

**결정**: 프로젝트 루트 `HANDOFF.md` [근거 강도: 중]

**근거**:
- `docs/history/index.md`의 "진행 중" 섹션이 현재 SSOT 역할을 하지만 (index.md L8: `SSOT: 이 섹션이 진행 중 정보의 유일한 원본`), 이 섹션은 장기 프로젝트 진행 상황용이고 단일 세션 인계서와는 목적이 다르다.
- `docs/history/HANDOFF.md`는 히스토리 디렉토리에 섞이므로 부적절. 인계서는 "완료된 기록"이 아니라 "임시 활성 정보"다.
- 루트 `HANDOFF.md`는 에이전트가 새 세션 시작 시 CLAUDE.md 다음으로 쉽게 감지 가능.
- `.gitignore` 대상 후보 (팀 공유 시에는 예외 처리).

**vs index.md 진행 중 섹션과의 관계** [근거 강도: 강]:
- index.md 진행 중 섹션은 **프로젝트 장기 SSOT** (여러 세션에 걸친 진행 상황, 7개 한계, 14일 한계 — index.md L7)
- `HANDOFF.md`는 **단일 세션 → 다음 세션 브리지** (다음 세션이 확인하면 즉시 소멸)
- 인계서 생성 시 index.md 진행 중 섹션도 함께 갱신하는 것이 바람직하다 (연계 정책 — 섹션 6 참조)

**파일 포맷**:
```markdown
# HANDOFF — {날짜} 세션 인계서

> 생성: YYYY-MM-DD HH:MM | 소멸 조건: 다음 세션 확인 후 `/handoff done`

## 마지막 상태 (어디까지 했나)
- 작업: [작업명]
- 진행률: [완료 단계/전체 단계]
- 마지막 편집 파일: `path/to/file` (L번호)

## 미완 작업 (지금 하다 멈춘 것)
- [ ] 항목 1 — 이유: 시간 부족 / 블로커 발생 / 결정 대기
- [ ] 항목 2

## 다음 세션 시작 지점
1. [명확한 첫 번째 행동 — 동사 시작]
2. [두 번째 행동]

## 미결 결정 (다음 세션에 결정 필요)
- 결정 사항: | 선택지: A / B | 현재 기울기: A

## 컨텍스트 (배경 이해용)
- 이 작업을 하는 이유: [간단히]
- 주의 사항: [있으면]

## 관련 파일
- `path/to/main_file` — 핵심 편집 대상
- `path/to/ref_doc` — 참조 문서
```

## 4. 인계 데이터 정의

인계서에 반드시 포함해야 할 정보:

| 데이터 | 설명 | 수집 방법 |
|--------|------|-----------|
| 마지막 상태 | 어디까지 완료했나 | git status + 대화 맥락 |
| 미완 작업 | 지금 하다 멈춘 것 (시급도 있음) | TaskList + 대화 맥락 |
| 다음 시작 지점 | 다음 세션의 첫 번째 행동 | 모델이 합성 |
| 미결 결정 | 다음 세션에 결정이 필요한 것 | 대화 맥락 |
| 관련 파일 | 핵심 편집 대상 경로 | git diff --name-only |

**수집 절차**:
1. `git status --short` + `git diff --stat` → 수정 파일 목록
2. `TaskList` → 미완 task 확인
3. index.md 진행 중 섹션 읽기 → 기존 장기 진행 중 항목 참조
4. 대화 맥락에서 미결 결정 추출 → HANDOFF.md에 정리

## 5. 다른 스킬과의 책임 경계

### vs /history (= /project-history) [근거 강도: 강]

| 구분 | /handoff | /history |
|------|----------|----------|
| 대상 | 미완 / 진행 중 | 완료된 것 |
| 지속성 | 임시 (다음 세션 확인 시 소멸) | 영구 (append-only 기록) |
| 작성 시점 | 세션 종료 직전 | 작업 완료 후 |
| 파일 | `HANDOFF.md` (루트) | `docs/history/{날짜}.md` |
| 용도 | 다음 세션 재개용 | 감사 추적, 레트로스펙트 |

**연계 정책**: `/handoff create` 시 `/history update`는 별도 호출. 인계서와 히스토리는 같은 시점에 동시 작업할 수 있지만 자동 연동은 하지 않는다 (사용자 제어 유지).

### vs /todo [근거 강도: 중]

| 구분 | /handoff | /todo |
|------|----------|-------|
| 발생 시점 | 세션 종료 (단절 시점) | 상시 |
| 미완 항목 성격 | "지금 하다 멈춘 것" (시급, 맥락 풍부) | "언제 할지 모르는 것" (백로그) |
| 소멸 조건 | 다음 세션 확인 후 즉시 | `done` 명령 또는 아카이브 |
| 수명 | 1~2 세션 | 프로젝트 존속 기간 |

**경계 명확화 규칙**: `/handoff` 미완 항목은 인계 완료 후 사용자 판단에 따라 `.todo.md`로 이동할 수 있다. 자동 이동 금지 — 인계 받은 세션이 항목의 우선순위를 재판단해야 하기 때문.

### vs index.md 진행 중 섹션 [근거 강도: 강]

| 구분 | /handoff (HANDOFF.md) | index.md 진행 중 섹션 |
|------|----------------------|----------------------|
| 범위 | 단일 세션 인계 | 프로젝트 장기 진행 상황 |
| 수명 | 1~2 세션 | 7개 한계·14일 한계까지 |
| 갱신 주체 | /handoff 스킬 | /history 스킬 또는 수동 |
| SSOT | 단절 세션 맥락 | 프로젝트 전체 진행 상황 |

**연계**: `/handoff create` 완료 시, index.md 진행 중 섹션도 갱신할 것을 권장(선택). index.md L8이 SSOT이므로 장기 진행 항목은 반드시 index.md에도 반영해야 이중화 방지.

## 6. 소멸 정책

**소멸 트리거**: 다음 세션에서 `/handoff done` 호출 시.

**소멸 절차**:
1. `HANDOFF.md` → `.backups/HANDOFF.done.{YYYY-MM-DD}.md`로 이동 (삭제 금지)
2. 미완 항목 중 지속 필요한 것 → 사용자에게 `.todo.md` 이동 여부 확인
3. index.md 진행 중 섹션 해당 항목 상태 업데이트

### (2) 반박/유보
**반박**: 섹션 3에서 `HANDOFF.md`를 루트에 두기로 했으나, index.md L8이 "SSOT: 이 섹션이 진행 중 정보의 유일한 원본"이라고 명시한다. 루트 `HANDOFF.md`를 별도로 두면 index.md SSOT 원칙과 충돌할 수 있다. 이 충돌은 "단일 세션 인계서"와 "장기 프로젝트 진행 상황"의 목적 차이로 정당화했지만, 실제 운영에서 두 곳에 비슷한 내용이 생기는 drift 위험은 남아있다.

### (3) 근거 강도
- 파일 위치(루트 HANDOFF.md): **중** (원리 기반, index.md와 차별화 논리)
- index.md 진행 중 섹션과의 관계: **강** (index.md 직접 인용)
- /history와 경계: **강** (project-history SKILL.md 직접 인용)
- /todo와 경계: **중** (원리 기반)

### (4) 자기 비판
이번 사양에서 놓쳤을 가능성: 인계서 "자동 생성" 시나리오가 없다. 세션이 비정상 종료되거나 사용자가 인계서 없이 세션을 끊는 경우, 다음 세션이 HANDOFF.md 없이 시작하는 문제가 미해결이다. 훅 기반 자동 생성(예: PreCompact 이벤트 활용)이 가능한지 검토가 필요하나 이번 사양에 포함하지 못했다.
